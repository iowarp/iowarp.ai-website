Platform Architecture

System internals, data flows, and component interactions in IOWarp

System Overview

IOWarp implements a three-tier hierarchical architecture designed for HPC-AI convergence at extreme scale. Each layer handles distinct concerns while maintaining tight integration with adjacent layers through well-defined interfaces and data contracts.

Layer 1: Intelligence Layer (CEI)

Purpose

The Intelligence Layer implements autonomous reasoning, natural language understanding, and agent orchestration. It's where human intent (natural language queries) gets transformed into structured computational actions.

Components

Natural Language Processing

  • Function: Parse user queries and scientific questions
  • Technologies: LLM integration, DSPy frameworks
  • Interfaces: User input → structured queries
  • Complexity: Semantic understanding of domain context

ReAct Agent Framework

  • Function: Implement reasoning and action loops for autonomous decision-making
  • Architecture:
    [Observation] → [Thought] → [Action] → [Reflection] → [Next Action]
  • Coordination: ClaudIO framework manages multi-agent orchestration
  • State Management: Maintains context and history across agent lifecycle

LLM Integration (IOWarp Interface)

  • Function: Domain-aware language model for scientific reasoning
  • Specialization: Science-specific knowledge, terminology, reasoning patterns
  • Integration Points: Agent decision-making, semantic enrichment, query formulation
  • Knowledge Injection: Research paper embeddings, domain ontologies
  • Primary Implementation: ClaudIO framework with DSPy programming

Context Bundle Reasoning

  • Function: Understand data relationships and scientific semantics
  • Data Model: Hierarchical, semantic context representation
  • Reasoning Engine: Determine optimal data access patterns and transformations
  • Output: Actionable instructions for Tool and Storage layers

Data Flow (Downward)

User Query (NL)
    ↓
[Parse & Contextualize]
    ↓
[ReAct Reasoning Loop]
    ↓
[Action Planning]
    ↓
[Tool Invocations + Storage Directives]
    ↓
Tool & Storage Layers

Data Flow (Upward)

Results from Tools & Storage
    ↓
[Process & Aggregate]
    ↓
[Context Update]
    ↓
[Next Reasoning Step or Return]
    ↓
User Response (Natural Language)

Layer 2: Tool Layer (CAE + PPI)

Purpose

The Tool Layer bridges the Intelligence Layer with the Storage Layer. It handles data ingestion, transformation, normalization, and extensibility through the plugin ecosystem.

MCP Ecosystem (15+ Servers):

  • SLURM job management
  • PBS/Torque integration
  • File I/O operations
  • Nextflow workflow execution
  • Parsl DAG execution
  • CWL workflow support
  • Scientific instrument control
  • Database query interfaces
  • Network protocols (SSH, SFTP, HTTP)
  • Custom domain tools
  • Data streaming management
  • Monitoring and telemetry
  • Version control integration
  • Configuration management
  • Extensibility framework

Layer 3: Storage Layer (CTE)

Purpose

The Storage Layer manages intelligent data movement, caching, and optimization across the heterogeneous storage hierarchy. It implements predictive prefetching and automated tiering.

Multi-Tier Storage Hierarchy

┌───────────────────────────────────────────────────────────┐
│           GPU Memory (VRAM)                               │
│    [Fastest, Most Limited]  ← On-Demand Staging          │
└───────────────────────────────────────────────────────────┘
                            ↑
                    [Hot Data Path]
                            ↑
┌───────────────────────────────────────────────────────────┐
│    Local NVMe Storage                                     │
│    [Fast, Limited Capacity] ← Intelligent Caching        │
└───────────────────────────────────────────────────────────┘
                            ↑
                    [Warm Data Path]
                            ↑
┌───────────────────────────────────────────────────────────┐
│    Parallel File System (Lustre, GPFS, etc.)             │
│    [Medium Speed, High Capacity] ← Working Set            │
└───────────────────────────────────────────────────────────┘
                            ↑
                    [Cold Data Path]
                            ↑
┌───────────────────────────────────────────────────────────┐
│    Object Storage / Archival                              │
│    [Slow, Unlimited] ← Long-term Storage                 │
└───────────────────────────────────────────────────────────┘
